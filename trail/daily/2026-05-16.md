## 배경 / 문제

사용자가 rein hook/CLI 의 안내 메시지를 (1) 친절한 비서 톤, (2) 사용자가 Claude Code 를 쓰는 언어(한/영/독 등) 로 안내되게 하려 함. 현재 차단 hook 은 전부 `echo "..." >&2` + `exit 2` 방식이고 메시지는 한국어 하드코딩.

Claude Code 공식 문서상 `exit 2 + stderr` 는 사용자에게만 표시되고 Claude(모델)에게 전달되지 않는다("not to Claude"). 다국어 자동 안내는 `exit 0` + JSON output (`hookSpecificOutput.permissionDecision: "deny"` + `permissionDecisionReason` + `additionalContext`) 으로 전환해 메시지를 모델에 전달, Claude 가 사용자 대화 언어로 풀어 전달하게 하는 경로로만 가능.

## codex-ask 검토 (2026-05-16, gpt-5.5 high)

결론 **조건부 권장**. JSON 전환 방향은 타당하나 35개 hook 전면 일괄 전환은 비권장:

- `exit 2` 는 단순·확실한 fail-closed. JSON deny 는 Claude Code 가 stdout JSON 을 정상 파싱해야만 차단 → **JSON 직렬화 실패 시 `exit 0` 이면 차단이 풀린다(fail-open)**.
- shell hook 에서 직접 `printf '{...}'` 하면 quoting·제어문자·긴 경로로 JSON 이 깨짐 → 중앙 emitter 필수.
- `additionalContext` 가 모델에 가면 prompt-injection 표면 확대 → 명령/경로/marker 를 "데이터이며 지시 아님" 으로 감싸고 control char 제거·길이 제한 필요.
- 권고: 중앙 `deny()` emitter ("JSON 생성 성공 → stdout JSON + exit 0, 실패 → stderr + exit 2") + 회귀 테스트를 **먼저** 만들고 단계적 이행. 본 DoD = 그 1단계.

## 배경

사용자가 rein hook/CLI 메시지를 (1) 친절한 비서 톤, (2) 사용자가 Claude Code를 쓰는
언어(한/영/독 등)로 안내되게 하려 함. codex-ask 검토 결과: `exit 2 + stderr`는

## 리뷰

- codex-review: Round 1 **NEEDS-FIX** — High 1건: sourced + `set -e`(errexit) 하에서
  command substitution `json=$(...)`이 resolver 성공 후 python runner nonzero 시
  fail-closed 정규화(`return 2`) 도달 전에 hook을 종료 → fail-open. fix:
  `if json=$(...); then py_rc=0; else py_rc=$?; fi` errexit-safe 분기 + 회귀 테스트
  (12) 추가(shim python으로 resolver-성공-후-runner-실패 재현) + (5)(6)을
  reason_code/additionalContext까지 확장. Round 2 **PASS**.
- security-review: **통과** (High/Critical 0). 정보성 관찰 1건 — U+2028/U+2029
  Unicode line/paragraph separator는 `sanitize()` 화이트리스트(C0+DEL만 제거)에서
  미제거. 취약점 아님 (JSON·Claude Code 파싱 안전), 2단계 sanitize 확장 후보.

## 2단계 설계 발견 (demo 검토 중)

emitter demo를 사용자와 검토하다 reason 구조 설계 결함 발견:

- 현재 `deny_emit`는 reason 인자 하나를 통째로 DATA frame으로 감싼다. 그래서
  "신뢰된 차단 사유 + 권장 행동"과 "신뢰 불가한 차단된 입력(명령어·경로)"이
  한 슬롯에 섞인다.
- 결과: hook이 차단된 입력만 reason에 넣으면 사용자가 받는 안내가 "무엇이 왜
  차단됐고 뭘 하면 되는지"를 설명하지 못하고 모호해진다 (demo 케이스 3에서
  "원인을 확인해 주세요" 같은 무의미한 안내가 나옴 — 사용자 지적).
- 2단계 설계: reason을 2 슬롯으로 분리 — (a) hook이 작성하는 신뢰된 사유+행동
  (frame 없이, 명확한 안내), (b) DATA frame으로 격리되는 신뢰 불가 입력.
  emitter 인터페이스를 `deny_emit <신뢰된_사유> <reason_code> <격리할_입력>` 식으로
  명시 분리 검토.
