# Workflow: fix-bug
> 버그 수정 절차. 증상 → 재현 → 원인 → 수정 → 검증 순서를 반드시 지킨다.

## 실행 절차

### Step 1: 버그 정보 수집
```
[ ] 버그 증상 명확히 기술
[ ] 재현 조건 파악 (입력값, 환경, 순서)
[ ] 영향 범위 파악
[ ] trail/incidents/ 유사 버그 전례 확인
```

### Step 2: 재현
```
[ ] 최소 재현 테스트 작성
[ ] 버그가 재현됨 확인 (테스트 실패 확인)
```

### Step 3: 원인 분석
```
[ ] 코드 흐름 추적
[ ] 원인 코드 라인 특정
[ ] 근본 원인(root cause) 명시 — 증상만 고치지 않는다
```

### Step 4: DoD 작성
```markdown
## DoD: [버그 설명]
- [ ] 재현 테스트가 통과함
- [ ] 근본 원인이 수정됨
- [ ] 기존 테스트 전체 통과
- [ ] 회귀 테스트 추가
- [ ] Self-review 완료
```

### Step 5: 수정
```
[ ] 최소한의 변경으로 수정
[ ] 테스트 통과 확인
```

### Step 6: Codex 코드 리뷰
```
[ ] /codex 스킬로 변경된 파일에 대해 리뷰 실행
[ ] 리뷰 결과의 수정사항 반영
[ ] 수정 후 테스트 재실행
```

### Step 7: 완료
```
[ ] trail/incidents/INC-NNN.md 작성
[ ] trail/index.md 갱신
```

## 금지
- 재현 없이 수정 시작 금지
- 증상만 숨기는 패치 금지 (try/catch로 삼키기 등)
